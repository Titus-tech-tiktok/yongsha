# Change2Pro 自适应批量生图设计

## 目标

修复批量生图时 30 张只成功约 4 张、其余图片因 `Upstream rate limit` 在三次重试后失败的问题。系统必须把 Change2Pro 的 50 路并发视为最大上限，通过自适应排队、错峰启动和限流退避逐步利用可用容量，而不是一次性向同一个图片 Key 发出 50 个请求。

本次同时修复 GitHub Actions 无法部署的问题，提供可核对的线上版本和有效运行参数。用户已经确认的 URL 图片响应、单次请求 300 秒超时、ZIP 下载流程和最终图片原尺寸规则保持不变。

## 已确认根因

当前调用链包含三层并发：

- 浏览器最多同时提交 50 个任务组。
- API 服务最多同时运行 50 个后台任务。
- 全局 Image2 信号量最多同时执行 50 个图片请求。

单个套图任务会由 API 服务将全部图片展开为独立图片请求。使用本地模拟 Change2Pro 限流的测试，30 张图片形成 30 路峰值并发；上游只接纳前 4 路时，结果为成功 4 张、失败 26 张，共发出 82 次请求，即 4 次成功加 26 张各 3 次失败请求。这与实际页面完全一致，说明系统不是只创建了 4 个任务，而是其余请求在 Change2Pro 网关或模型账户池阶段被拒绝。

现有重试会继续占用全局图片槽位，并让大量失败请求在相近时间再次发出。它没有读取 `Retry-After`，也不会根据 429/5xx 反馈降低并发，因此会形成重试拥塞。

GitHub Actions 的测试命令为 `node --test tests`。该写法在当前 Windows 环境可以运行，但在 GitHub 的 Linux Node.js 24 环境中把 `tests` 当作模块路径并报 `Cannot find module .../apps/api/tests`，导致所有自动部署均跳过。

## 推荐方案

采用单 Key、全局自适应图片调度器。调度器管理所有账号和任务的 Image2 请求，配置上限为 50，但运行并发会根据上游反馈动态调整。

核心规则：

- 最大并发固定为 `50`，可通过 `CAISHEN_IMAGE_API_MAX_CONCURRENCY` 调低，不允许配置超过 50。
- 冷启动并发为 `4`，可通过 `CAISHEN_IMAGE_API_INITIAL_CONCURRENCY` 配置，但不能超过最大并发。
- 新请求按固定最小间隔启动，默认每 `500ms` 最多启动一个，避免冷启动瞬间形成 30 到 50 路上传洪峰。
- 每完成 10 次请求形成一个健康窗口；窗口成功率至少 90% 且没有限流错误时增加 2 路并发，直至达到 50。
- 收到 429、408、425、502、503、504 或包含 `Upstream rate limit`、`temporarily unavailable` 的响应时，将运行并发减半，最低降到 1。
- 优先读取响应的 `Retry-After`；没有该字段时使用以 1 秒为基数、最长 120 秒的指数退避加全随机抖动。
- 退避期间释放运行槽位，将请求重新放回队列，不能让等待中的失败请求占住 50 个槽位。
- 临时错误最多重试 8 次。重试期间图片状态为“等待上游恢复”，不能提前计入永久失败；达到上限才记录最终失败。
- 用户点击“立即停止”时，队列中和退避中的该任务请求都必须取消，正在上传的请求通过 `AbortSignal` 中止。

## 大图传输

当前实现每次请求都读取并上传原始套图和原始印花。同一张印花会在一个 30 张套图任务中重复上传 30 次。真实素材达到数 GB 时，这会同时放大服务器内存、磁盘读取、出口带宽和 Change2Pro multipart 解析压力。

在调用 Image2 前增加只读参考图缓存：

- 以源文件绝对路径、文件大小、修改时间和转换规格生成缓存键。
- 最长边超过 2048 像素时等比例缩小到 2048，禁止放大较小图片。
- 保留透明通道；透明图片输出压缩 PNG，非透明图片输出质量 92 的 JPEG。
- 同一任务中的印花只转换一次，所有图片请求复用缓存文件。
- 缓存只作为 API 参考输入，不覆盖用户上传的原图，也不作为最终下载文件。
- 缓存文件大小和原文件大小写入诊断日志，用于确认传输缩减比例。

## 图片尺寸

API 生图尺寸根据套图原图方向选择 Change2Pro Image2 支持的最接近规格：

- 接近正方形：`1024x1024`
- 横图：`1536x1024`
- 竖图：`1024x1536`

生成结果下载后继续由 Sharp 输出为套图原图的精确像素宽高。比例不一致时使用 `cover` 居中裁剪，不增加白边。ZIP 中只包含该最终文件，因此参考图预处理和 URL 下载都不能改变 ZIP 图片的目标尺寸、命名或质量规则。

## 可观测性

每次图片请求记录不含密钥的结构化事件：

- 任务和套图相对路径
- 请求尝试次数
- 排队、开始、响应和完成时间
- HTTP 状态码和 Change2Pro 返回的错误摘要
- 当时的运行并发、并发上限和队列长度
- 原始参考图总字节数和缓存参考图总字节数
- API 返回 URL 的下载耗时

任务进度区显示“运行中、等待上游、已完成、永久失败”四类数量。Change2Pro 临时限流时，用户能够看到系统正在降速重试，而不是直接得到 26 张失败。

新增无需登录的 `/api/health` 最小健康接口，返回应用提交版本、启动时间、最大并发、当前自适应并发、活动请求数和队列长度，不返回 API Key、文件路径或用户数据。该接口用于确认 GitHub 部署后服务器确实运行目标提交。

## 部署修复

将 API 测试脚本改为 Node.js 在 Windows 和 Linux 都能识别的测试文件 glob，并在本地 Windows 与 GitHub Linux Node.js 24 环境验证。

部署流水线必须满足：

1. 安装依赖。
2. 构建 Web 和 API。
3. 运行全部测试。
4. 只有测试通过才通过 SSH 更新服务器。
5. 重建容器后请求健康接口，校验返回的提交 SHA 等于本次部署 SHA。
6. 健康检查失败时让部署任务失败，不能把未启动的新容器标记为部署成功。

## 测试与验收

自动测试必须覆盖：

- 模拟上游最多同时接纳 4 路请求时，30 张图片最终全部成功，不出现 26 张永久失败。
- 实际同时处于上游处理状态的请求不超过调度器当前并发。
- 429 后并发降低，连续成功后逐步恢复，但永远不超过 50。
- `Retry-After` 优先于本地退避时间。
- 退避中的请求不占用活动槽位。
- 点击停止能够取消排队、退避和正在请求的图片。
- 参考图缓存按源文件变化失效，同一印花在同一批次复用缓存。
- 横图、竖图和方图选择正确 API 规格，最终文件严格等于原套图像素宽高且没有白边。
- URL 返回图片能够继续预览，并按套图相对路径和文件名进入任务 ZIP。
- GitHub Linux 测试命令能够发现并运行全部测试文件。

线上验收分两轮：

1. 使用 30 张套图执行一次批量任务，确认任务日志存在 30 个图片请求，临时限流会排队恢复，最终完成数明显高于当前 4 张。
2. 逐步扩大到 100、500 张，观察稳定并发、平均耗时、限流比例、服务器内存和出口流量，再决定是否提高冷启动并发。

## 产能边界

50 是单个 Change2Pro 图片 Key 的最大允许并发，不等于一天必然能完成 10 万张。若单张平均耗时为 60 到 90 秒，理论上 50 路全天满载约为 4.8 万到 7.2 万张。要达到每天 10 万张，平均耗时必须低于 43.2 秒，或者后续增加第二个独立 Image2 Key 和第二组调度容量。

本次先实现单 Key 稳定调度，不引入 Redis、多服务器 worker 或多 Key 轮换。单机稳定性验证完成后，再单独设计 10 万张规模的分布式任务系统。

## 不在本次范围

- 多个 Image2 Key 的轮换、权重和余额感知调度
- Redis/BullMQ 分布式任务队列
- 多台生图 worker 自动扩缩容
- 修改 Change2Pro 模型分组或账户池配置
- 修改现有美元计费规则
