# 套图换印花分析与任务输出设计

日期：2026-07-14

## 背景

当前系统把“图片是否需要出现在成品中”和“图片是否需要调用生图 API”混合在一个动作字段里。Web 端又只把 `replace_print` 图片加入任务，导致 `copy_template` 虽然在后端可以直接复制，实际创建任务时却没有进入任务范围。

同时，系统允许没有有效标注区域的图片保存为 `replace_print`。现有 UI 会用“运营确认的留白家具面板”等泛化文案替代失败分析，使页面看起来已经配置，但没有可执行的精确边界。生成阶段只上传模板图和印花图，没有使用已保存蒙版，因此提示词中的“只修改柜门”无法成为硬约束。

## 目标

1. AI 先判断图片是否需要保留在成品套图中，再判断是否存在可印花的白色家具外表面。
2. 需要保留但不能换印花的图片，按原目录和原文件名逐字节复制到每个新任务中，不调用生图 API。
3. `replace_print` 必须具有明确的面板描述和非空有效蒙版；否则只能进入人工确认。
4. 印花作为一张连续完整画面跨所有可印花白色柜门，保持原始颜色和宽高比，完整缩放到可印花区域内，不裁切、不拉伸、不重复、不重排、不越界。
5. 印花比例与柜面比例不一致时，未覆盖区域保留模板原有白色柜面。
6. 模板中的文字、人物、背景、墙地面、柜体结构、门缝、把手、边框、柜脚、道具和阴影不被生成模型改写。
7. 所有校验必须发生在计费请求之前。

## 非目标

1. AI 不负责重新设计印花或自动挑选原印花中的局部元素。
2. AI 不得自动删除套图文件夹中的图片。
3. 本次不改变任务 ZIP 的目录结构和文件命名规则。

## 状态模型

业务上分开记录两个维度：

- `includeInOutput`：图片是否进入最终任务文件夹。
- `processingMode`：`replace_print`、`copy_original` 或 `manual_check`。

Web 端可以继续显示四个动作，但语义固定为：

| 显示动作 | includeInOutput | processingMode | 行为 |
| --- | --- | --- | --- |
| 换印花 | true | replace_print | 使用精确蒙版迁移印花 |
| 保留原图 | true | copy_original | 直接复制原文件，不调用 API |
| 人工确认 | 未确定 | manual_check | 阻止创建任务，等待运营处理 |
| 排除该图 | false | 无 | 不生成、不复制，仅允许运营手动选择 |

原“不处理”统一改名为“排除该图（不输出）”。AI 不允许自动选择排除；无法套印花时默认保留原图。

## AI 分析决策顺序

AI 必须按以下顺序判断，不允许跨步：

1. 判断图片在套图中的用途，例如主图、场景图、尺寸图、细节图、物流包装、安装售后或纯文字说明页。
2. 判断图片是否需要保留在最终套图中。套图文件夹中的图片默认需要保留。
3. 判断目标家具是否清晰可见。
4. 判断是否存在清晰、可见、属于家具外侧的白色或浅色柜门、抽屉正面或面板。
5. 没有可印花表面时选择保留原图；无法判断内外侧、遮挡或边界时选择人工确认。
6. 只有存在明确可印花表面时选择换印花，并为每个面板输出精确区域。

物流、包装、安装、售后、纯文字页、没有目标家具的必要详情页，以及只展示侧面、背面、内部但没有可印花白色外表面的图片，均为保留原图。

## AI 分析契约

分析结果至少包含：

```json
{
  "version": 9,
  "imageRole": "主图",
  "includeInOutput": true,
  "processingMode": "replace_print",
  "confidence": 0.96,
  "imageUnderstanding": "正面场景主图，黑色四门柜，四扇闭合白色柜门外表面可印花。",
  "viewState": "正面闭合",
  "printableSurfaces": [
    {
      "id": "front-door-1",
      "label": "左起第一扇白色柜门外表面",
      "polygon": [[0.16, 0.47], [0.32, 0.47], [0.32, 0.78], [0.16, 0.78]],
      "surfaceState": "外侧闭合"
    }
  ],
  "mappingMode": "continuous_across_surfaces",
  "preserveAreas": "文字、背景、黑色柜体、边框、门缝、把手、柜脚、道具和阴影",
  "riskPoints": ["金色把手位于面板内，必须从蒙版中扣除"]
}
```

`replace_print` 的输出必须满足：

- `includeInOutput=true`。
- `printableSurfaces` 至少包含一个有效多边形。
- 每个多边形坐标均在 0 到 1 内，面积达到最低阈值且不越出图片。
- 不得把墙面、地面、文字、人物、标签、门缝、把手、柜脚、黑色边框、柜门内侧或柜体内部包含在蒙版中。
- 多宫格图片必须分别描述每个小图的可印花面板；任何小图不确定时整张进入人工确认。

服务端必须使用严格结构校验。未知动作、缺字段、空区域、低置信度、区域异常或 JSON 不完整一律降级为人工确认，不能默认成换印花。

## Web 分析结果界面

三个文本框调整为：

1. “图片用途与判断”：显示事实性图片理解和选择该动作的原因。
2. “可印花区域”：逐项描述可印花柜门/抽屉外表面及其开合状态。
3. “必须保持不变”：列出文字、背景、结构、五金和遮挡物。

换印花时必须同时显示蒙版叠加预览、面板数量和覆盖比例。不存在蒙版时，保存按钮禁用并提示“请先标注可印花区域”。系统不得再自动填入“运营确认的留白家具面板”来伪装有效分析。

保留原图时：

- 可印花区域显示“无”。
- 必须保持不变显示“整张原图”。
- 页面明确提示“将原文件直接复制到新任务，不调用 API”。

## 印花迁移规则（方案 A）

1. 将所有可印花外表面视为同一幅连续画布，而不是每扇门重复一张印花。
2. 保持原印花的 RGB 内容、元素相对位置、方向和宽高比。
3. 使用 `contain` 规则把完整印花居中缩放到可印花区域，不裁切、不拉伸、不平铺。
4. 印花未覆盖到的柜面继续使用模板原有白色像素。
5. 印花只在蒙版内可见；蒙版外强制恢复模板原始像素。
6. 门缝、把手、边框和前景遮挡物位于印花上层。
7. 对无法可靠建立透视映射的开门、半开、多平面或严重遮挡图片，停止处理并转人工确认，不允许模型猜测。

为了满足“不改色、不重排、完整迁移”，普通换印花流程采用确定性像素贴合，而不是让生成模型重新绘制印花。AI 只负责理解图片和提出候选区域；运营可修正蒙版。生图 API 仅保留给明确需要生成式修复的特殊流程，不作为普通印花贴合的默认实现。

## 任务创建与输出

创建一个套图任务时，任务计划自动包含：

- 运营选中的所有换印花图片。
- 同一套图文件夹内全部保留原图图片。
- 不包含排除图片。
- 只要存在人工确认图片，就阻止任务启动并给出文件列表。

任务卡显示完整计划，例如：

`输出 33 张：换印花 5 张 + 原图复制 28 张 + 排除 0 张`

后端按原相对路径写入结果。`copy_original` 使用文件复制，不转码、不调整尺寸、不改变质量。任务 ZIP 继续保持任务名作为 ZIP 名，内部目录和文件名与套图文件夹一一对应。

## 失败与计费保护

以下情况必须在调用付费接口前失败：

- 换印花但没有有效蒙版。
- 蒙版覆盖禁止区域或面积异常。
- 印花文件不存在或无法读取。
- 模板分析仍为人工确认。
- 映射平面不明确。

任务失败信息必须说明具体图片和失败校验，不得自动回退到无蒙版生图。原图复制失败应单独报告为文件系统错误，不得转为生图请求。

## 验收标准

1. 没有蒙版的图片不能保存为换印花，也不会产生费用。
2. 物流图和无白色柜面详情页在新任务中与模板原文件逐字节相同。
3. 排除图片不会进入任务文件夹。
4. 同一印花在不同套图图片中保持完整内容、颜色和元素顺序，不被重新设计。
5. 蒙版外模板像素保持不变，文字和人物不被修改。
6. 印花比例不一致时完整显示，剩余区域保持原白色。
7. 任务卡和 ZIP 文件数量等于换印花数量加保留原图数量。
8. 人工确认未处理完时不能启动任务。
